# 5.3.2　安全功能模块设计（SOC 对齐 · 四类机制）

本节与 SOC 指挥中心（`/security`）四张指标卡一一对应，覆盖源码注释中的 **「安全模块 1～4」**。统一出口为 **`security_logs`** 结构化落库 **`event_type`** 字段，再由 **Realtime · `postgres_changes`** 增量刷新计数，从而形成「纵深防御 → 离散事件 → 态势可视」闭环。

---

### （一）安全模块 1：URL 矢量 XSS 边缘拦截（`middleware.ts`）

**目标**：在进入应用路由与应用逻辑前，阻断 **反射型类 XSS** 通过 **`pathname + search`** 混入。  
**机制**：请求进入 **Edge Middleware** 后，`detectXss` 在 **`decodeURIComponent`** 解码后的字符串上执行关键字正则命中（如 `script`、`javascript:`、事件处理器等），必要时对原始字符串回退匹配以降低编码绕过成功率。命中后 **`await insertEdgeSecurityAudit`** 写入 **`XSS Attack`**，并返回品牌化 **403 HTML**（示例头 **`X-Blocked-By: secure-trade-edge-waf`**，`Cache-Control: no-store`），避免恶意页被 CDN 长效缓存。**局限声明**：单层编码绕过仅为课堂演示级可控面，生产需叠加 **CSP、输出编码、商业 WAF 规则簇**。

---

### （二）安全模块 2：登录暴力破解限速（同上 · `consumeRateLimit`）

**目标**：压低 **口令喷洒（Password Spraying）** 在单位时间内的试错吞吐。  
**机制**：针对 **`POST /api/auth/**`**，`handleAuthRateLimit` 依据 **`LOGIN_RATE_LIMIT`**（默认 **60 s / 最多 5 次**），以 **`auth:${客户端IP}`** 为桶键调用 **`consumeRateLimit`**。**允许**时在响应头注入 **`X-RateLimit-*`**；**拒绝**时返回 **`429`** 与 **`Retry-After`**，并写入 **`Brute-force Attempt`** 审计行。**拓扑说明**：当前为 **单机进程 `Map`** 近似滑动计数，多实例漂移需替换为 **Redis / Upstash** 等价共享令牌桶。

---

### （三）安全模块 3：密码学与认证失败可追溯（`app/api/auth/login`）

**目标**：口令 **不以明文入库**，认证失败可被 SOC 归因而不泄露过高隐私粒度。  
**机制**：载荷先经 **`Zod`** 规整邮箱与小幅度长度约束 **`password`**；核心业务调用 **`supabase.auth.signInWithPassword`**——口令校验与 **bcrypt 风格摘要比对**托管于 **Supabase Auth / GoTrue**（服务端实现细节以托管版本为准）。**失败路径**异步调用 **`recordAuthFailureAudit`**，向 **`security_logs`** 写入 **`Auth Failure`**，**payload 仅含掩码邮箱**（示例：保留首尾字符，`@` 后域名完整可读）， **`defense_level: api`**。成功路径则将 **HttpOnly Cookie 会话** 通过 **`attachAuthCookies`** 附加到 **`NextResponse`**，避免凭证滞留于前端可读存储。**与模块 2 关系**：限速先消耗「次数预算」，摘要比对再消耗「正确性」，两类事件可分开展示 SOC 热力。

---

### （四）安全模块 4：API 契约与白名单校验 / 注入预防（`createApiHandler` + `recordSchemaValidationAudit`）

**目标**：把一切进入业务逻辑的 JSON **先投影到 Zod Schema**，从而在类型与格式层消灭 **大面积注入试探与字段混淆**。  
**机制**：`createApiHandler` 在 **`await request.json()`** 后执行 **`safeParse`**；失败时 **`await recordSchemaValidationAudit`**（工程上需 **await** 防止 Serverless 冻结导致「422 却无行」），写入 **`Schema Validation`**，**payload** 截取 **`pathname`** 与 **首条 issue 路径**，避免整块 body 灌水。校验通过才进入 **`checkout`**、实验路由等 **`handler`**。结合 **`checkout.safeText`** 等 refine，可把 **HTML 片段注入订单展示名**的风险前移到 **422**。结合 **Postgres 参数化**，形成 **契约层 + 语句层** 双闸。
